# GUI 中文化升级方案

> 日期：2026-03-28
> 状态：已归档
> 对应实施计划：`docs/superpowers/plans/2026-03-28-gui-chinese-localization-plan.md`

## 背景

当前 OpenClaw Node 的桌面 GUI 已完成基础功能，但仍存在较多英文用户文案，影响中文场景下的使用一致性。同时，仓库尚未统一“代码注释使用中文”的约束，后续迭代容易出现中英混杂和维护成本上升的问题。

本次升级以“用户可见内容中文化”为目标，补齐 GUI 文案和注释规范，不调整既有功能与技术架构。

## 目标

1. 将 GUI 页面中的主要用户可见文案统一调整为中文。
2. 建立代码注释使用中文的规范，优先覆盖高价值位置。
3. 不影响现有 Wails、React、Go 后端绑定结构。
4. 为后续引入 i18n 保留清晰的演进路径。

## 范围

### 本次包含

1. 导航、页面标题、副标题、表单标签、按钮、空状态、提示信息中文化。
2. 状态展示、能力展示、操作反馈的中文化映射。
3. 在关键逻辑处补充中文注释。
4. 建立集中式文案管理方式。

### 本次不包含

1. 不引入完整 i18n 框架。
2. 不修改变量名、类型名、接口字段名、路由路径、协议命令字。
3. 不翻译原始技术日志、协议事件名和结果 JSON 字段。
4. 不做页面结构和交互流程重设计。

## 总体策略

本次采用“展示层中文化、内部标识保持英文”的策略。

核心思路如下：

1. 前端新增统一文案模块，集中管理中文展示文本。
2. 页面和组件不再散写英文硬编码，统一从文案模块取值。
3. 状态值、能力名、提示信息通过映射表输出中文。
4. 内部逻辑判断仍使用英文枚举和值，避免影响现有实现。
5. 中文注释只用于解释复杂意图、兼容边界和映射原因，不做无意义重复注释。

## 影响范围

主要影响前端展示层：

1. `frontend/src/components/layout/*`
2. `frontend/src/pages/*`
3. `frontend/src/components/shared/*`
4. `frontend/src/hooks/*`
5. `frontend/src/lib/*`

仅当后端直接返回英文展示值且前端无法稳定映射时，才考虑在 `internal/wails/*` 做最小补充。

## 注释规范

代码注释统一使用中文，并遵循以下原则：

1. 注释解释“为什么这样做”，不复述代码表面行为。
2. 对状态映射、展示兜底、协议透传、技术英文保留原因等复杂点写中文注释。
3. 简单 JSX、样式拼接、直观赋值不加注释。

## 里程碑

### 阶段 1：文案收口

完成导航、页头、通用按钮和通用提示的中文集中定义。

### 阶段 2：页面中文化

完成 Dashboard、Connection、Capabilities、Operations、Logs、About 页面的静态文案替换。

### 阶段 3：动态反馈中文化

完成状态展示、Toast、能力状态、空状态和结果面板的中文映射。

### 阶段 4：收尾与验证

完成中文注释补充、构建验证、测试验证和残留英文登记。

## 风险

1. 后端若直接返回英文展示值，前端可能需要补额外映射。
2. 技术日志和原始结果数据不适合强行翻译，需要明确边界。
3. 如果中文文案再次散落到组件中，后续接入 i18n 会增加返工成本。

## 后续演进

后续若需要引入 i18n，可在当前集中式文案结构基础上继续演进：

1. 将中文文案拆为 `zh-CN` 资源。
2. 增加 `en-US` 等语言资源。
3. 接入翻译函数替换当前直接取值方式。

因此，本次方案既满足当前中文化目标，也不会阻碍未来的多语言扩展。
